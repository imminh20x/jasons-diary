# Quality Control (QC) Agent Rules & Guidelines

This document outlines standard E2E testing directives, locator hierarchies, and verification guidelines for the **Quality Control (QC)** subagent or test engineers working on this repository.

---

## 1. 10 Core Principles of Playwright Best Practices

All generated E2E testing suites in this project must adhere to the official TestDino-mandated E2E principles:

1. **Prefer Accessible Selectors**: Locate elements by semantic role (`getByRole`), text labels (`getByLabel`), or placeholders (`getByPlaceholder`) rather than CSS classes or raw HTML structures.
2. **Never Use Hard Timeouts**: Do not use `page.waitForTimeout()`. Wait for network conditions, element visibilities, or route-specific operations instead.
3. **Use Web-First Assertions**: Always use auto-retryable assertions like `expect(locator).toBeVisible()` rather than checking visibility properties procedurally.
4. **Isolate Every Test**: Ensure test flows do not share state, cache side-effects, or require sequential execution order to pass.
5. **Centralize Base URLs**: Always utilize relative paths (e.g. `page.goto('/')`) in tests, setting the host in `playwright.config.ts` via `baseURL`.
6. **Robust CI configuration**: Run tests with `retries: 2` in CI to handle environmental instability, and `retries: 0` locally for rapid development.
7. **Trace Collection**: Set trace collection to `'on-first-retry'` to preserve system memory while capturing rich traces for debug analysis.
8. **Share State via Fixtures**: Use custom Playwright fixtures or Page Object Models (POM) rather than defining module-level global variables.
9. **Single Behavior Scope**: Each test block (`test('should...', async () => {})`) must verify one logical behavior or flow.
10. **Mock External Integrations**: Mock external APIs (e.g., Supabase authentication, cloud services) to isolate app behavior, but never mock your own client application code.

## 2. Playwright Locator Hierarchy

To build resilient E2E tests and ensure high accessibility standards, prioritize selectors according to the following strict hierarchy:

| Priority | Locator API | Best Used For |
| :--- | :--- | :--- |
| **1. Semantic Role** | `page.getByRole('role', { name: 'label' })` | Interactive elements (buttons, links, textboxes, headings) |
| **2. Text Label** | `page.getByLabel('Text label')` | Form inputs associated with `<label>` tags |
| **3. Placeholder** | `page.getByPlaceholder('...')` | Search bars or fields with placeholder text |
| **4. Explicit Text/Alt** | `page.getByText('Text')`, `page.getByAltText('Alt')` | General content blocks, visual images with descriptive text |
| **5. Test ID** | `page.getByTestId('test-id')` | Complex components lacking semantic roles or unique content |

### Forbidden Locators
- **No Style CSS Classes**: Avoid targeting classes like `page.locator('.btn-primary')` or auto-generated Tailwind styles.
- **No Fragile XPath / CSS Selector Chains**: Avoid structures like `page.locator('div > div > span > button')` which break instantly on design changes.

## 3. Auditing & Quality Checks

In addition to user flow checks, E2E tests must include the following quality verification gates:

### A. Console & Network Diagnostics
- **Unhandled Exceptions**: Monitor browser console logs during test execution. Tests must fail if unhandled JavaScript errors occur in the client console.
- **Network Failures**: Check network responses. No 404 or 500 status codes should be generated by assets, layout fonts, or API requests.

### B. Visual Regression Check
## 4. Visual Regression & Layout QA
- Maintain reference screenshots of stable states (e.g., Homepage desktop view, Blog post hero + TOC split layout).
- Verify layout consistency using `expect(page).toHaveScreenshot()` when styling or layout changes are made.
- Post detail pages must align horizontally with other `.container` pages (`--layout-max`); prose column width is computed beside the 300px TOC sidebar.
- Post back navigation: `data-testid="link-back-articles"` in the hero (link, not button in body).

### C. Accessibility (a11y) Audits
- Integrate `@axe-core/playwright` to run accessibility audits on key templates (e.g., check for color contrast, proper landmark structures, keyboard focus traps, and WCAG 2.1 compliance).
